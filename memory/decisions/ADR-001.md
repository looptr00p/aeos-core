# Architecture Decision Record

**Traceability ID**: ADR-001

## Title

Dual Directory Structure — Root Governance + Experimental Lab Subproject

## Status

PROPOSED

## Date

2026-05-19

## Owner

architect-agent (proposed), human operator (approval required)

## Review Date

Pending

## Context

AEOS Core operates as a governance-first operating system where the repository is the single source of truth. During the repository truth reconciliation audit (2026-05-19), a structural inconsistency was identified: the README.md and ESTADO_DEL_ARTE.md repository tree diagrams show all directories under `aeos-core/`, but the actual repository has these directories at the root level (`governance/`, `protocols/`, `templates/`, `agents/`, `workflows/`, `memory/`, `scripts/`, `tests/`, `docs/`).

The `aeos-core/` subdirectory exists as a nested subproject containing experimental lab artifacts (experiments, porting policy, experimental templates, enforcement tests). This creates a dual-directory system that is not documented, not governed by explicit decision, and causes confusion in onboarding, CI configuration, and traceability.

The CI workflow (`.github/workflows/aeos-validation.yml`) runs `pytest aeos-core/tests`, which executes only 3 enforcement tests and misses the 11 root-level test files that validate governance, protocols, templates, agents, and workflows. This means the core governance validation suite is not executed in CI.

## Problem

AEOS Core has two parallel directory structures without an explicit architectural decision governing their relationship:

1. **Root level** — Contains the canonical AEOS Core governance, protocols, templates, agents, workflows, memory, and the primary test suite (11 files).
2. **`aeos-core/` subdirectory** — Contains an experimental lab with its own governance (porting policy), experiments, experimental templates, and enforcement tests (3 files).

This creates the following operational problems:

- **Onboarding confusion**: New contributors cannot determine which directory structure is canonical.
- **CI gap**: CI runs only `aeos-core/tests`, missing the root test suite that validates governance compliance.
- **Documentation drift**: README.md tree diagram is incorrect, creating a false mental model.
- **Lint duplication**: Two `aeos_lint.py` scripts exist (root and aeos-core) with unclear delegation boundaries.
- **Traceability ambiguity**: It is unclear whether `aeos-core/` memory artifacts participate in the main traceability index.

## Constraints

- AEOS Core must remain markdown-first and repo-driven.
- The experimental lab must remain isolated from canonical governance (experiments are not canonical by default).
- No autonomous runtime may be introduced.
- Porting from experimental lab to canonical base requires explicit review and human approval (per PORTING_POLICY.md).
- All governance validation must be executable and verifiable.
- The repository must remain the single source of truth.
- Changes must not break existing operational memory artifacts.
- The solution must be maintainable by a single human operator.

## Options Considered

### Option 1: Flatten — Remove aeos-core/ Subdirectory

Merge all experimental lab artifacts into the root directory structure. Experiments would live alongside canonical artifacts, distinguished by naming convention (EXP- prefix, AUD-EXP- prefix).

- **Pros**: Single directory structure eliminates confusion. CI runs all tests. Simpler mental model. No delegation needed for lint scripts.
- **Cons**: Blurs the boundary between experimental and canonical artifacts. Increases risk of accidental porting of unreviewed experiments. Violates the explicit separation stated in README.md (lines 167-173). Requires restructuring 15+ experimental memory artifacts.
- **Reason Rejected**: The experimental lab's purpose is to test governance changes in isolation before porting. Flattening removes this isolation boundary, increasing the risk of unreviewed experimental changes being treated as canonical. This violates the PORTING_POLICY.md principle that "no experiment may be ported automatically."

### Option 2: Invert — Move Everything Under aeos-core/

Move all root-level directories into `aeos-core/`, making it the canonical location. Update CI, lint scripts, and all references.

- **Pros**: Matches the current README.md tree diagram. Single canonical location. CI path becomes correct without modification.
- **Cons**: Breaks all existing file references in memory artifacts (53 files reference root-level paths). Requires updating 14 test files. Requires updating lint scripts. High migration risk. Creates a nested path (`aeos-core/governance/`) that is less ergonomic for daily operations.
- **Reason Rejected**: The migration cost is prohibitive. 53 memory artifacts reference root-level paths. 14 test files use root-level BASE_DIR calculations. The risk of breaking traceability during migration outweighs the benefit of matching the incorrect README diagram.

### Option 3: Dual Structure with Explicit Governance (Recommended)

Keep both directory structures but formalize the relationship through explicit documentation, CI configuration, and governance rules. Root is canonical. `aeos-core/` is the experimental lab. CI runs both test suites. Documentation is corrected.

- **Pros**: Preserves experimental isolation. Fixes CI gap. Corrects documentation. No migration of existing artifacts. Clear governance boundary. Maintains porting workflow.
- **Cons**: Two directory structures to maintain. Requires explicit documentation of the relationship. Requires CI configuration change. Slightly more complex mental model (but documented).
- **Reason Rejected**: Not rejected — this is the recommended option.

### Option 4: Separate Repositories

Move the experimental lab to a separate repository. Keep AEOS Core as a single-purpose governance repository.

- **Pros**: Cleanest separation. No structural ambiguity. Each repository has a single purpose.
- **Cons**: Loses the porting workflow (cross-repository porting is more complex). Requires managing two repositories. Breaks the "repo-driven" principle for the experimental lab. Increases operational overhead for a single human operator.
- **Reason Rejected**: The experimental lab exists to test changes that may be ported to the canonical base. A separate repository makes porting more complex and loses the git-traceable relationship between experiment and canonical artifact. The single-repository dual-structure approach is simpler for a single operator.

## Decision

**Option 3: Dual Structure with Explicit Governance**

The repository will maintain two directory structures with explicitly governed boundaries:

1. **Root directories** (`governance/`, `protocols/`, `templates/`, `agents/`, `workflows/`, `memory/`, `scripts/`, `tests/`, `docs/`) are the **canonical AEOS Core** governance framework. All operational artifacts, traceability IDs, and governance documents live here.

2. **`aeos-core/` subdirectory** is the **AEOS Experimental Governance Lab**. It contains:
   - Experimental governance documents (e.g., PORTING_POLICY.md)
   - Experiment artifacts (EXP-XXX, PORT-XXX)
   - Experimental templates (experiment_template.md, porting_review_template.md)
   - Enforcement tests for experimental lab validation
   - Delegated lint script that validates both root and experimental requirements

3. **CI configuration** will be updated to run both test suites:
   - `pytest tests/` — canonical governance validation
   - `pytest aeos-core/tests/` — experimental lab enforcement

4. **Documentation** will be corrected to reflect the actual structure and explicitly document the dual-directory relationship.

5. **Porting** from `aeos-core/` to root requires explicit review, validation evidence, and human approval per PORTING_POLICY.md.

## Rationale

This decision preserves the experimental lab's isolation while fixing the operational problems caused by undocumented dual structure. The key principles driving this decision are:

- **Isolation over simplicity**: Keeping experimental artifacts separate from canonical governance prevents accidental adoption of unreviewed changes. The porting workflow provides a controlled migration path.
- **Explicit over implicit**: The dual structure is not inherently problematic — the problem is that it was undocumented. Making it explicit resolves confusion without restructuring.
- **CI completeness**: Running both test suites ensures that governance validation is not silently skipped. The current CI gap is a governance risk.
- **Low migration cost**: This option requires no migration of existing artifacts. Only documentation, CI configuration, and lint scripts need updates.
- **Single-operator maintainability**: A separate repository (Option 4) would increase overhead for the current single human operator. The dual structure within one repository is manageable.

## Consequences

### Positive

- CI validates the complete governance test suite (11 root tests + 3 experimental tests = 14 tests).
- Documentation accurately reflects repository structure, enabling correct onboarding.
- Experimental isolation is preserved with explicit governance boundary.
- Porting workflow remains functional and auditable.
- No existing operational memory artifacts are disrupted.
- Traceability IDs remain valid and unchanged.
- The lint delegation pattern (aeos-core lint → root lint) is documented and intentional.

### Negative

- Two directory structures must be maintained and understood by contributors.
- CI workflow is slightly more complex (two test invocations).
- The `aeos-core/` prefix in paths is redundant for the experimental lab (it is "aeos-core within aeos-core").
- Contributors must learn the distinction between canonical and experimental directories.

### Tradeoffs

| Tradeoff | Decision | Rationale |
|----------|----------|-----------|
| Isolation vs. simplicity | Isolation preferred | Prevents accidental adoption of unreviewed experimental changes |
| Single repo vs. separate repos | Single repo | Lower operational overhead for single operator; preserves git-traceable porting |
| CI complexity vs. coverage | Coverage preferred | Missing governance validation is a higher risk than CI complexity |
| Documentation accuracy vs. migration cost | Documentation update preferred | Fixing docs is lower risk than restructuring 53 memory artifacts |

## Risks

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| CI workflow change breaks test execution | MEDIUM | Low | Validate both test paths locally before committing; run full test suite |
| Contributor modifies experimental artifacts thinking they are canonical | LOW | Medium | Explicit documentation of boundary; PORTING_POLICY.md enforcement |
| aeos-core/ lint script drifts from root lint script | LOW | Medium | Document delegation relationship; periodic audit |
| Future porting decisions become ambiguous | LOW | Low | PORTING_POLICY.md provides clear criteria; this ADR reinforces it |

## Validation Strategy

| Validation | Method | Expected Result |
|-----------|--------|-----------------|
| README tree accuracy | Manual comparison vs actual directory listing | Tree matches repository structure |
| CI test execution | `pytest tests/ && pytest aeos-core/tests/` | All 14 tests pass |
| Lint delegation | `python3 aeos-core/scripts/aeos_lint.py` | Delegates to root lint, passes |
| Experimental isolation | Verify no root-level artifact references aeos-core/ memory IDs | No cross-contamination |
| Porting policy enforcement | Verify PORT-001.md status | Porting remains deferred |
| Documentation consistency | grep for `aeos-core/` references in root docs | All references are intentional and documented |

## Future Reassessment Conditions

This ADR will be reassessed when any of the following conditions occur:

1. **Phase transition to Phase 2**: Multi-project usage may require separate repositories for canonical and experimental concerns.
2. **Contributor count exceeds 3**: Multiple operators increase the risk of confusion between canonical and experimental directories.
3. **Experimental lab produces 10+ portable artifacts**: High porting volume may justify a more structured migration workflow.
4. **CI execution time exceeds 60 seconds**: Dual test suite execution may require optimization or parallelization.
5. **Porting policy is invoked**: Any actual porting event triggers a review of whether the dual structure remains appropriate.
6. **Repository size exceeds 500 files**: Structural complexity may warrant reorganization.
