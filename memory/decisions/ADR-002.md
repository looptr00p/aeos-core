## Context

AEOS Core has comprehensive governance documentation but weak operational enforcement. The governance review (REV-010) identified that human approval is required but not recorded, ADR requirements exist but are not validated, lifecycle transitions are not tracked, and CI does not enforce governance integrity. The gap between declared governance and operational enforcement undermines AEOS's core value proposition: governance-first, auditable, human-controlled AI-assisted engineering.

The deterministic validation review (VAL-001) confirmed that 89.8% of repository references are consistent, but 12 checks failed due to path inconsistencies, version mismatches, and CI configuration gaps. These are detectable issues that should be caught automatically.

## Problem

Governance rules exist as documentation but are not operationally enforceable. Specifically:

1. **No approval trail**: Human approval is required for critical changes but no artifact records that approval occurred. There is no way to audit whether a governance change was human-approved.
2. **No ADR enforcement**: ADR_PROTOCOL.md requires ADRs for governance, architecture, CI, permission, and protocol changes. No automated check validates this requirement.
3. **No lifecycle transition tracking**: Tasks move between states (DRAFT → IN_PROGRESS → REVIEW → CLOSED) without recording when, by whom, or whether prerequisites were met.
4. **No CI governance validation**: CI runs file existence tests but does not check governance compliance (approval records, ADR presence, lifecycle validity).
5. **No orphan detection**: Artifacts can reference non-existent objectives, tasks, or other IDs without detection.
6. **No stale objective detection**: Objectives can remain ACTIVE indefinitely without progress, degrading governance continuity.
7. **No escalation aging**: Escalations persist indefinitely without automatic aging or escalation to human.
8. **No version enforcement**: Three artifacts declare different versions with no mechanism to detect or prevent inconsistency.

Without deterministic enforcement, governance compliance depends on manual review, which is error-prone and non-reproducible.

## Constraints

- Must preserve human authority as final decision maker — enforcement checks inform, they do not decide.
- Must remain markdown-first — all governance artifacts are human-readable markdown.
- Must remain repository-driven — no external databases or state stores.
- Must remain locally executable — no cloud dependencies or runtime services.
- Must remain deterministic — all checks produce the same result given the same input.
- Must not introduce autonomous systems — no self-approving or self-modifying behavior.
- Must not add hidden state — all enforcement state is visible in git history.
- Must remain maintainable by a single human operator — complexity must be bounded.
- Must not break existing functionality — all current tests and lint checks must continue to pass.

## Options Considered

### Option 1: Inline Lint Checks Only

Extend `scripts/aeos_lint.py` with all governance enforcement checks. No new artifact types. No CI changes beyond running the existing lint.

- **Pros**: Minimal complexity. Single script to maintain. No new directories or templates. Existing workflow unchanged.
- **Cons**: Lint checks can detect violations but cannot record approvals. No approval trail is created. Lifecycle transitions are not tracked. Enforcement is limited to what can be checked against file content.
- **Reason Rejected**: Insufficient. Cannot address the approval record gap or lifecycle transition tracking. Lint-only approach detects problems but does not create the artifacts needed for auditability.

### Option 2: Full Governance Engine

Create a Python-based governance engine that parses all artifacts, validates compliance, tracks lifecycle transitions, manages approval workflows, and produces governance reports.

- **Pros**: Comprehensive enforcement. Automated lifecycle tracking. Approval workflow management. Governance health reporting.
- **Cons**: High complexity. Introduces runtime behavior. Creates hidden state (engine internal state). Violates "no orchestration" principle. Requires significant maintenance. Single-operator burden increases dramatically. Risk of framework inflation.
- **Reason Rejected**: Violates core AEOS principles. Introduces orchestration, hidden state, and runtime complexity. Over-engineered for the problem.

### Option 3: Deterministic Enforcement Layer (Recommended)

Extend the existing lint script with governance enforcement checks. Add approval record artifacts (APR-XXX) as a new artifact type. Add lifecycle transition logging to task files. Update CI to run governance checks. All enforcement is file-based, deterministic, and auditable.

- **Pros**: Preserves all AEOS principles. Low complexity (~200 lines of new lint code). No runtime behavior. All state is visible in git. Human authority preserved. Maintainable by single operator. Composable — checks can be added or removed independently.
- **Cons**: Approval records must be created manually (not automated). Lifecycle transitions must be logged manually. Enforcement is limited to what can be expressed as file-based checks.
- **Reason Rejected**: Not rejected — this is the recommended option.

### Option 4: GitHub-Enforced Governance

Use GitHub branch protection rules, required reviewers, and environment protection to enforce governance at the platform level.

- **Pros**: Platform-level enforcement. No additional code. Human approval is enforced by GitHub UI.
- **Cons**: Ties AEOS to GitHub. Not locally executable. Not repository-driven (depends on external platform). Not auditable from repository alone. Violates "repo-driven" principle.
- **Reason Rejected**: Violates repository-driven principle. AEOS governance should be enforceable from the repository alone, independent of hosting platform. GitHub protection can complement but not replace repository-level enforcement.

## Decision

**Option 3: Deterministic Enforcement Layer**

AEOS governance enforcement will be implemented through three complementary mechanisms:

1. **Extended lint checks** (`scripts/aeos_lint.py`): 10 new deterministic checks that validate governance compliance by inspecting repository files. All checks are file-based, deterministic, and locally executable.

2. **Approval record artifacts** (APR-XXX): New artifact type that records human approval decisions. Stored in `memory/approvals/`. Created manually by the approving human or by an agent documenting a human's approval. Template-based and auditable.

3. **Lifecycle transition logging**: Embedded in task files as a simple table. Records when a task moved between states, by whom, and with what justification. No new artifact type — additive field in existing task template.

CI will be updated to run the extended lint script. The lint output will include governance enforcement results alongside existing checks.

## Rationale

This decision balances enforcement capability with operational simplicity. The key principles are:

- **Detection over prevention**: Lint checks detect governance violations. They do not prevent changes. Human authority remains the decision maker.
- **Explicit over implicit**: Approval records are explicit artifacts. No hidden approval state. No implicit approval inference.
- **File-based over runtime**: All enforcement state is stored in files. No runtime engine. No database. No hidden state.
- **Composable over monolithic**: Each lint check is independent. Checks can be added, removed, or modified without affecting others.
- **Manual over automated**: Approval records are created manually. Lifecycle transitions are logged manually. Automation is limited to detection, not action.

The lint-based approach is sufficient for the current scale of AEOS (66 canonical artifacts, 7 agents, 14 tests). If AEOS scales beyond 200 artifacts, a more structured approach may be needed, but that is a future concern.

## Consequences

### Positive

- Governance compliance is detectable and auditable.
- Human approval decisions are recorded as explicit artifacts.
- Lifecycle transitions are tracked in task files.
- CI validates governance integrity on every push/PR.
- No runtime complexity or hidden state introduced.
- All enforcement is locally executable and repository-driven.
- Single-operator maintainability is preserved.
- Existing functionality is not broken.

### Negative

- Approval records must be created manually, adding a small workflow step.
- Lifecycle transition logging requires manual entry at each state change.
- Lint checks are limited to what can be expressed as file-based validation.
- Some governance rules (e.g., "human approval required") cannot be fully enforced — only the presence of an approval record can be checked, not the authenticity of the human's decision.
- False positives are possible if lint checks are not carefully designed.

### Tradeoffs

| Tradeoff | Decision | Rationale |
|----------|----------|-----------|
| Detection vs. prevention | Detection preferred | Prevention requires runtime enforcement; detection is sufficient for auditability |
| Manual vs. automated approval records | Manual preferred | Automated records would create hidden state; manual records are explicit and auditable |
| Lint vs. governance engine | Lint preferred | Engine introduces runtime complexity; lint is file-based and deterministic |
| GitHub vs. repository enforcement | Repository preferred | GitHub ties AEOS to a platform; repository enforcement is platform-independent |
| Comprehensive vs. minimal checks | Minimal preferred (10 checks) | Comprehensive checks increase maintenance burden; minimal checks address the highest-impact gaps |

## Risks

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| Lint check false positives | LOW | Medium | Test all checks against known-good state before deployment |
| Approval records become bureaucratic overhead | LOW | Low | Records are lightweight (5 fields); not a bottleneck |
| Lifecycle transition logging is incomplete | LOW | Medium | Missing transitions are detectable by lint; gaps flagged |
| Enforcement checks cannot verify human authenticity | MEDIUM | High | Checks verify record existence, not authenticity. Human authenticity is a social trust issue, not a technical one. |
| Lint complexity grows unmanageable | LOW | Low | Cap at 15 checks total; current plan is 21 (11 existing + 10 new) — will consolidate existing checks to stay under 15 |
| CI enforcement blocks legitimate changes | LOW | Low | Enforcement checks are informational first; blocking requires explicit configuration |

## Future Reassessment Conditions

This ADR will be reassessed when any of the following conditions occur:

1. **Artifact count exceeds 200**: Current lint-based approach may become unwieldy at scale.
2. **Multiple human operators**: Manual approval records may need workflow automation.
3. **False positive rate exceeds 10%**: Lint checks need refinement or removal.
4. **Governance rules change significantly**: Enforcement checks must be updated to match.
5. **AEOS moves to Phase 2 or beyond**: Multi-project usage may require platform-level enforcement.
6. **Lint check count exceeds 15**: Must consolidate or split into multiple scripts.
