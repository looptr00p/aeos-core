## Context

AEOS Core has comprehensive governance documentation (7 governance files, 7 protocols, 11 templates, 6 workflows, 7 agents) and an enforcement layer design (OBJ-007, ADR-002). However, AEOS cannot observe how governance behaves under operational pressure. From OBJ-004 and OBJ-005, we know that reviewer contention, escalation accumulation, and recovery asymmetry are real phenomena. But these observations are qualitative — we cannot measure them systematically, track them longitudinally, or predict their behavior.

The governance review (REV-010) scored AEOS at 1.8/4.0. The deterministic validation (VAL-001) confirmed 89.8% consistency. These are point-in-time measurements. AEOS needs the ability to track governance health over time, understand pressure accumulation patterns, and study organizational dynamics — without introducing automation, databases, or runtime complexity.

## Problem

AEOS governance operates under organizational dynamics that are currently unmeasured and untrackable:

1. **No governance metrics**: We know governance pressure exists but cannot quantify it.
2. **No longitudinal data**: All observations are point-in-time. We cannot track trends or detect drift.
3. **No simulation capability**: We cannot model how governance behaves under stress scenarios.
4. **No operational dashboard**: Governance health is scattered across multiple artifacts with no unified view.
5. **No research framework**: We cannot systematically study organizational dynamics or governance behavior patterns.

Without observability, governance improvements are based on intuition rather than measurement.

## Constraints

- Must remain markdown-first — all metrics, dashboards, and reports are human-readable markdown
- Must remain repository-driven — all data stored in git-versioned files
- Must remain deterministic — metric calculations produce the same result given the same input
- Must remain locally executable — no cloud dependencies or runtime services
- Must remain lightweight — measurement overhead must not exceed 10% of operational effort
- Must preserve human authority — metrics inform decisions, they do not make them
- Must not introduce hidden state — all measurement data is visible in repository
- Must be maintainable by a single human operator
- Must not modify existing governance documents

## Options Considered

### Option 1: Metrics-Only Approach

Define governance metrics but do not create simulation scenarios, dashboards, or longitudinal tracking.

- **Pros**: Minimal complexity. Metrics are defined and calculable from repository state.
- **Cons**: Metrics without context are not actionable. No simulation capability. No longitudinal tracking. No unified view of governance health.
- **Reason Rejected**: Insufficient. Metrics alone do not provide observability — they need context (simulations), presentation (dashboards), and time dimension (longitudinal tracking).

### Option 2: Full Observability Platform

Create a Python-based observability platform that automatically collects metrics, generates dashboards, runs simulations, and produces reports.

- **Pros**: Comprehensive observability. Automated metric collection. Real-time dashboards. Predictive modeling.
- **Cons**: High complexity. Introduces runtime behavior. Creates hidden state (platform internal state). Violates "no orchestration" principle. Requires significant maintenance. Single-operator burden increases dramatically. Risk of framework inflation.
- **Reason Rejected**: Violates core AEOS principles. Introduces orchestration, hidden state, and runtime complexity. Over-engineered for the problem.

### Option 3: Markdown-First Observability Framework (Recommended)

Define metrics with explicit formulas. Design simulation scenarios based on operational experience. Create a markdown dashboard template. Produce observation reports manually. All data is stored in repository files. All calculations are deterministic and manually verifiable.

- **Pros**: Preserves all AEOS principles. Low complexity. No runtime behavior. All state is visible in git. Human authority preserved. Maintainable by single operator. Composable — metrics and scenarios can be added independently. Enables longitudinal research.
- **Cons**: Metrics must be calculated manually (or via simple scripts). Dashboard is regenerated per observation cycle, not real-time. Simulation scenarios are models, not executable simulations.
- **Reason Rejected**: Not rejected — this is the recommended option.

### Option 4: External Observability Service

Use an external service (e.g., Grafana, Datadog) for governance observability.

- **Pros**: Professional dashboards. Real-time monitoring. Alerting. Historical data.
- **Cons**: Ties AEOS to external platform. Not locally executable. Not repository-driven. Introduces hidden state (service internal data). Violates "repo-driven" principle. Requires infrastructure.
- **Reason Rejected**: Violates repository-driven principle. AEOS observability should be self-contained within the repository.

## Decision

**Option 3: Markdown-First Observability Framework**

AEOS governance observability will be implemented through four complementary mechanisms:

1. **Governance metrics framework** — 12 metrics with explicit formulas, thresholds, and operational interpretations. Each metric is calculable from existing repository state.

2. **Pressure simulation scenarios** — 8 operational stress models based on real operational experience (OBJ-004, OBJ-005). Scenarios describe conditions, expected outcomes, and validation criteria.

3. **Markdown dashboard template** — A reusable template that presents governance health metrics in a unified view. Regenerated per observation cycle.

4. **Operational observation reports** — Periodic reports (OPR-XXX) that document governance health trends, pressure patterns, and longitudinal observations.

All observability data is stored in repository files. All calculations are deterministic. No automation, databases, or runtime complexity is introduced.

## Rationale

This decision balances observability capability with operational simplicity. The key principles are:

- **Measurement over automation**: We measure governance behavior; we do not automate governance decisions.
- **Explicit over implicit**: All metric formulas are documented. No hidden calculations. No black-box analytics.
- **File-based over runtime**: All observability data is stored in files. No runtime engine. No database. No hidden state.
- **Composable over monolithic**: Each metric, scenario, and dashboard section is independent. Components can be added or removed without affecting others.
- **Manual over automated**: Metrics are calculated manually (or via simple scripts). Dashboards are regenerated per cycle. Automation is limited to calculation, not interpretation.

The markdown-based approach is sufficient for the current scale of AEOS (66 canonical artifacts, 7 agents, 14 tests). If AEOS scales beyond 200 artifacts, a more structured approach may be needed, but that is a future concern.

## Consequences

### Positive

- Governance behavior is measurable and trackable over time.
- Operational pressure patterns are understood and modeled.
- Longitudinal governance research becomes possible.
- Governance health is visible in a unified dashboard view.
- No runtime complexity or hidden state introduced.
- All observability is locally executable and repository-driven.
- Single-operator maintainability is preserved.
- Existing functionality is not broken.

### Negative

- Metrics must be calculated manually or via simple scripts, adding operational overhead.
- Dashboard is not real-time — it reflects the state at the time of report generation.
- Simulation scenarios are descriptive models, not executable simulations.
- Longitudinal analysis requires manual data collection across observation cycles.
- Some metrics may be difficult to calculate precisely from repository state alone.

### Tradeoffs

| Tradeoff | Decision | Rationale |
|----------|----------|-----------|
| Measurement vs. automation | Measurement preferred | Automation would violate human authority; measurement informs decisions |
| Manual vs. automated calculation | Manual preferred (with simple scripts) | Automated calculation creates hidden state; manual calculation is explicit |
| Descriptive vs. executable simulations | Descriptive preferred | Executable simulations require runtime; descriptive models are sufficient for understanding |
| Markdown vs. platform dashboards | Markdown preferred | Platform dashboards tie AEOS to external services; markdown is repository-driven |
| Comprehensive vs. minimal metrics | Minimal preferred (12 metrics) | Comprehensive metrics increase overhead; minimal metrics address highest-impact governance dimensions |

## Risks

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| Metrics become measurement overhead | MEDIUM | Medium | Cap at 12 metrics; each must be calculable in <5 minutes |
| Simulation scenarios are unrealistic | LOW | Medium | Base scenarios on actual operational data |
| Dashboard template becomes stale | LOW | Low | Dashboard is regenerated per cycle, not maintained continuously |
| Longitudinal analysis requires more data than available | MEDIUM | High | Start with available data; identify gaps for future collection |
| Metrics interpreted as governance automation | LOW | Low | Explicit documentation that metrics inform, not decide |
| Operational entropy indicators are too abstract | LOW | Medium | Ground each indicator in observable repository state |

## Future Reassessment Conditions

This ADR will be reassessed when any of the following conditions occur:

1. **Artifact count exceeds 200**: Current markdown-based approach may become unwieldy at scale.
2. **Multiple human operators**: Manual metric calculation may need automation.
3. **Longitudinal data exceeds 12 observation cycles**: Data management may need structured approach.
4. **Governance rules change significantly**: Metrics must be updated to match.
5. **AEOS moves to Phase 2 or beyond**: Multi-project usage may require cross-project observability.
6. **Metric count exceeds 15**: Must consolidate or prioritize metrics.
