## Context

AEOS internal agents (analyst, architect, engineer, qa, reviewer) produce measurable, testable, and auditable specifications from OBJ-XXX objectives. Today those specifications reach external code generators directly without passing through a deliberate translation layer. This produces inconsistent prompts, insufficient context, and low-quality outputs from external generators (Claude, GPT, etc.).

A Prompt Engineer role exists implicitly in the operator's workflow — it was externalized as a GPT with its own knowledge base (`gpt_knowledge_base/prompt_engineering/`). With `gpt_knowledge_base/` removed (ADR-001 consequence, Phase 0 closure), that external artifact no longer exists in the repository. The role must be formalized as an internal AEOS agent.

## Decision

Add `prompt-engineer-agent` as a first-class AEOS agent with `DOCS_ONLY` scope.

The Prompt Engineer Agent receives specifications already processed by internal agents (with acceptance criteria, technical constraints, and delimited scope) and produces structured prompts with complete context bundles for external code generators.

Key constraints of the decision:
- The PE does NOT receive raw OBJ-XXX from the Director — only specs processed by at least one internal agent
- The PE does NOT write code, define features, or make architectural decisions
- The PE does NOT approve its own work
- The PE cannot modify governance/, protocols/, workflows/, or agents/
- The PE can be invoked from any node in the cognitive graph when specs are sufficiently mature

## Alternatives Considered

### Alternative 1: Keep PE External (GPT / external tool)

- **Pros**: No repository changes. No new agent to maintain.
- **Cons**: External GPT has no access to AEOS memory, traceability context, or governance constraints. Prompts produced without AEOS context are inconsistent. Removing gpt_knowledge_base/ eliminates the only shared knowledge between AEOS and the external GPT. The gap between internal specs and external generation becomes ungoverned.
- **Reason Rejected**: The external GPT depended on gpt_knowledge_base/ which is now removed. Keeping the PE external without the shared knowledge base would produce worse outputs than before, and the role would be completely ungoverned.

### Alternative 2: Distribute PE Responsibilities Across Existing Agents

Give each existing agent the ability to produce prompts for external generators as part of their scope.

- **Pros**: No new agent. Prompt production stays close to spec production.
- **Cons**: Prompt engineering is a specialized skill — it requires understanding of model context windows, prompt structure, and external generator behavior. Distributing it across all agents dilutes the specialization and makes prompt quality inconsistent. No single agent owns quality of external generation.
- **Reason Rejected**: Specialization is a core AEOS principle. Prompt engineering requires dedicated focus. A distributed approach produces the same inconsistency problem the PE is designed to solve.

### Alternative 3: Dedicated Prompt Engineer Agent (Recommended)

- **Pros**: Explicit position in the cognitive graph. Single owner of external prompt quality. Clear input/output contract (receives processed specs, produces TASK-XXX context bundles). Auditable via memory/tasks/ and memory/reviews/. Preserves AEOS traceability.
- **Cons**: One more agent to maintain. PE must be invoked deliberately — does not automatically receive work.
- **Reason Rejected**: Not rejected — this is the accepted option.

## Consequences

- External code generation is now governed and traceable within AEOS
- TASK-XXX artifacts produced by the PE are auditable and linked to the specs that originated them
- The Director has a protocol for when and how to engage the PE (DIRECTOR_PROTOCOL.md)
- Prompt quality review reports (memory/reviews/) provide longitudinal quality tracking
- The PE can be invoked multiple times per work cycle — no fixed position in a pipeline
- If a spec is insufficient for prompt production, the PE escalates to the originating agent (not the Director)
