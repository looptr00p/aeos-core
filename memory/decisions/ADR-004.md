# Architecture Decision Record

**Traceability ID**: ADR-004

## Title

AEOS Evolution: OpenCode Integration, State Graph Orchestration, and AI Engineering Studio Direction

## Status

PROPOSED

## Date

2026-05-20

## Owner

director-agent (proposed), human operator (approval required)

## Review Date

Pending

## Context

AEOS Core v0.3 established a solid governance-first framework with 8 governance files, 7 protocols, 6 workflows, 12 templates, 7 agents, and validation tools. However, it operates in Phase 0 (Governance/Bootstrap) without integration to any execution layer.

A strategic conversation with the operator revealed that the core problem AEOS solves is not "how to create agents" but "how to preserve strategic and operational continuity while multiple intelligences participate in a project." This reframes AEOS from a multi-agent framework to **organizational cognition infrastructure**.

The operator's original problem: "I became the messenger between IAs" — manually routing context between multiple agents for planning, prompts, and coding.

## Problem

AEOS Core v0.3 has three gaps that prevent it from being operational:

1. **No execution layer integration**: Agents are defined but not connected to any LLM execution platform (OpenCode, Claude Code, Cursor, etc.)
2. **Linear workflows**: Current workflows are sequential pipelines, but real development has bidirectional feedback loops (QA finds bugs → developer fixes → reviewer re-reviews → auditor finds governance gaps → etc.)
3. **Not template-izable**: AEOS is a specific repo, not a reusable template that can be applied to any project

## Options Considered

### Option 1: Build from scratch as new project

Create a new "AI Engineering Studio" project separate from AEOS Core.

- **Pros**: Clean slate, no legacy constraints
- **Cons**: AEOS v0.3 governance is solid and validated. Rebuilding would waste months of work. Loses the existing memory, ADRs, and validation suite.
- **Reason Rejected**: AEOS already has the governance layer that is the hardest part to build. Evolution is more efficient than replacement.

### Option 2: Evolve AEOS Core with layered architecture (Recommended)

Keep AEOS Core as the governance foundation and add layers on top:
- Layer 1 (existing): Cognitive Governance Layer
- Layer 2 (new): Agent Orchestration Layer (OpenCode integration)
- Layer 3 (future): Execution Layer (agnostic to LLM provider)

- **Pros**: Preserves existing governance work. Adds execution capability incrementally. Maintains markdown-first, repo-driven philosophy.
- **Cons**: Requires careful evolution to not break existing governance. More complex than greenfield.
- **Reason Rejected**: Not rejected — this is the recommended option.

### Option 3: Integrate with existing multi-agent framework

Adopt an existing framework (LangGraph, AutoGen, CrewAI) and adapt AEOS governance to it.

- **Pros**: Existing orchestration engine, community support
- **Cons**: These frameworks are pipeline-oriented, not governance-first. They introduce hidden state, runtime orchestration, and autonomous behavior — all forbidden by AEOS safety rules. Would require fundamentally changing AEOS philosophy.
- **Reason Rejected**: Violates AEOS core principles (no hidden memory, no autonomous execution, no runtime orchestration). AEOS governance must precede execution, not adapt to execution frameworks.

## Decision

**Option 2: Evolve AEOS Core with layered architecture**

The evolution will proceed in phases:

### Phase 1: OpenCode Integration (Capa 2 — Agent Orchestration)
- Create `.opencode/opencode.json` with 7 agents and mixed models
- Create agent definition files in `.opencode/agents/`
- Create `aeos-governance` skill for OpenCode
- Models: claude-sonnet-4 for director/architect (strategic reasoning), claude-3.5-haiku for execution agents (cost efficiency)

### Phase 2: State Graph Orchestration
- Create `workflows/state_graph.md` defining nodes, edges, and feedback loops
- Create `protocols/STATE_TRANSITION_PROTOCOL.md` for recording transitions
- Create `memory/state-log/` for transition records
- Update all existing workflows to reference state graph and feedback loops

### Phase 3: Template-ization (Future)
- Create reusable AEOS template structure
- Adapt governance to be project-agnostic
- Create `project-charter.md` template

### Phase 4: CLI `studio` (Future)
- Build CLI tool for bootstrapping, managing, and auditing AEOS projects
- Commands: `studio init`, `studio status`, `studio audit`, `studio validate`

## Rationale

This decision preserves AEOS's governance-first philosophy while adding the execution capability needed to make it operational. The layered architecture ensures that:

- **Governance is permanent**: Layer 1 (governance) survives changes to layers 2 and 3
- **Agents are replaceable**: Layer 2 (orchestration) can be swapped (OpenCode → other platform)
- **LLMs are replaceable**: Layer 3 (execution) can use any provider (Anthropic → OpenAI → others)
- **Memory stays in the project**: All artifacts remain markdown files in the repo

The state graph approach reflects reality better than linear pipelines: development has loops, not straight lines.

## Consequences

### Positive

- AEOS becomes operational, not just theoretical
- Agent definitions are now executable through OpenCode
- Workflows reflect real development patterns with feedback loops
- Mixed model approach optimizes cost (expensive models only where needed)
- Governance skill ensures all agents follow AEOS rules
- State transition logging provides audit trail of workflow execution

### Negative

- Increased complexity: 7 agent definitions + governance skill + state graph
- OpenCode dependency: requires OpenCode to execute (but agents are platform-agnostic in design)
- Learning curve: operators must understand state graph, not just linear workflows
- Token cost: multiple agent invocations per cycle (mitigated by mixed models)

### Tradeoffs

| Tradeoff | Decision | Rationale |
|----------|----------|-----------|
| OpenCode-specific vs. platform-agnostic | OpenCode first, design for portability | Need execution now; agent definitions are markdown and portable |
| 7 agents vs. fewer agents | 7 agents for full coverage | Each agent has distinct cognitive responsibility; consolidation would blur boundaries |
| Mixed models vs. single model | Mixed models | Cost optimization: strategic reasoning needs powerful models, execution does not |
| State graph vs. pipeline | State graph | Real development has feedback loops; pipelines are unrealistic |

## Risks

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| OpenCode integration doesn't work as expected | MEDIUM | Low | Agent definitions are markdown and portable; can adapt to other platforms |
| State graph is too complex for simple projects | MEDIUM | Medium | Create "AEOS Lite" profile with simplified graph for small projects |
| Token cost exceeds budget | MEDIUM | Medium | Mixed models already optimize cost; monitor and adjust |
| Governance skill conflicts with OpenCode's built-in skills | LOW | Low | AEOS governance skill is project-specific; OpenCode skills are general |
| State transition logging creates overhead | LOW | Medium | Automate transition logging through agent prompts |

## Validation Strategy

| Validation | Method | Expected Result |
|-----------|--------|-----------------|
| OpenCode config validity | `opencode` starts without errors | Config loads, agents available |
| Agent definitions | Each agent file loads in OpenCode | All 7 agents recognized |
| Governance skill | Skill triggers on AEOS work | Governance rules enforced |
| State graph | Execute full cycle with feedback loops | All transitions logged in `memory/state-log/` |
| Workflow updates | `aeos_lint.py` passes | All workflow files valid |
| Mixed models | Verify model assignments in opencode.json | Director/architect use sonnet-4, others use haiku |

## Future Reassessment Conditions

This ADR will be reassessed when:
1. **Phase 1 validated**: First operational cycle through OpenCode completes successfully

3. **Phase 3 begins**: Template-ization work starts
4. **Token cost exceeds threshold**: Monthly cost analysis shows optimization needed
5. **Platform migration considered**: Need to support execution platforms beyond OpenCode
6. **Multi-project usage**: AEOS applied to 3+ different projects
